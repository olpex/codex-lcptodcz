# UX/UI беклог для GitHub Issues (готово до вставки)

Використання:
1. Скопіюй один блок `Issue`.
2. Створи нову задачу в GitHub Issues.
3. Встав у Description, а Title/Labels задай з блоку.

Рекомендовані лейбли:
- `ux`
- `ui`
- `frontend`
- `priority/P0` або `priority/P1` або `priority/P2`
- `effort/S` або `effort/M` або `effort/L`

## Issue 1

Title: `UX-001: Прибрати автозаповнення admin-логіну/паролю на сторінці входу`
Labels: `ux`, `ui`, `frontend`, `priority/P0`, `effort/S`, `security`

Проблема:
Форма логіну містить передзаповнені значення, що небезпечно і вводить в оману в продакшені.

Мета:
Сторінка входу відкривається з порожніми полями логіну і паролю.

Scope:
- Прибрати дефолтні значення `admin`/пароль у стані форми.
- Залишити лише placeholders.
- Не змінювати валідацію та API-виклик логіну.

Acceptance Criteria:
- При першому відкритті поля порожні.
- У UI не відображається жоден дефолтний пароль.
- Логін працює з валідними обліковими даними.

Out of Scope:
- Зміни бекенд-автентифікації.
- SSO/LDAP інтеграції.

Тест-план:
- Відкрити `/login`, перевірити порожні поля.
- Увійти валідними даними.
- Перевірити повідомлення для невалідного логіну.

## Issue 2

Title: `UX-002: Винести аварійне скидання паролю в окремий флоу`
Labels: `ux`, `ui`, `frontend`, `priority/P0`, `effort/M`, `security`

Проблема:
Аварійне скидання паролю змішане з основним логіном і перевантажує екран входу.

Мета:
Розділити стандартний логін і аварійне відновлення доступу на окремі сценарії.

Scope:
- Прибрати reset-форму з основного екрану логіну.
- Додати дію `Аварійне відновлення доступу`.
- Створити окрему сторінку або модальне вікно для reset token + new password.

Acceptance Criteria:
- На сторінці логіну лише форма входу.
- Аварійний reset доступний як вторинна дія.
- Поточна логіка reset працює без регресій.

Out of Scope:
- Повна переробка auth backend.
- MFA.

Тест-план:
- Перевірити, що login-screen не містить reset-полів.
- Перевірити роботу аварійного сценарію end-to-end.

## Issue 4

## Issue 7

Title: `UX-007: Уніфікувати форми (label, helper text, inline validation, loading state)`
Labels: `ux`, `ui`, `frontend`, `priority/P1`, `effort/M`, `a11y`

Проблема:
Форми місцями тримають логіку в placeholders і не дають однакового фідбеку.

Мета:
Передбачувані, доступні форми з однаковим UX.

Scope:
- Видимі labels для всіх бізнес-критичних полів.
- Helper text для складних полів.
- Єдиний формат inline-помилок.
- Єдиний disabled/loading стан кнопки submit.

Acceptance Criteria:
- Немає ключових інпутів без label.
- Помилки показуються біля відповідного поля.
- Кнопка відправки має консистентний loading.

Out of Scope:
- Переписування форм на іншу бібліотеку.

Тест-план:
- Перевірка keyboard-only сценарію.
- Базова перевірка screen reader.

Labels: `ux`, `ui`, `frontend`, `priority/P2`, `effort/L`, `a11y`, `design-system`

Проблема:
Компоненти і стилі від модуля до модуля виглядають непослідовно.

Мета:
Єдина візуальна мова і передбачувана поведінка інтерфейсу.

Acceptance Criteria:
- Основні сторінки використовують спільні примітиви.

Out of Scope:

Тест-план:
- Lighthouse accessibility прогін.
- Ручний keyboard-only аудит ключових сторінок.
